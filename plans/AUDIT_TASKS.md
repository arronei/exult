# Exult - Code Audit Task List

Full-codebase bug audit (604 C++ files / ~238k lines) across 22 graphify-first review units. Each finding was self-verified by the reviewing agent (offending line read and quoted). The **6 critical** findings were additionally hand-verified against the source. High/medium/low are agent-verified but not individually re-checked - confirm before fixing.

**Totals:** 6 critical | 19 high | 17 medium | 6 low  (48 tasks)

## CRITICAL

- [x] **[gumps/Gump.cc:218](gumps/Gump.cc#L218)** - Gump::find_object() writes into a fixed-size stack array with no bounds check, causing a stack buffer overflow if more than 100 objects in the container overlap the mouse point. `[buffer-overflow]` **[hand-verified]** — **FIXED, build-verified**
    - **Problem:** A container with more than 100 objects stacked at the same tile (e.g. a large reagent/food pile, or items created via cheat commands) is opened in a gump; clicking anywhere over the stacked pile makes find_object() iterate all matching objects and write past list[99], corrupting the stack (cnt, s, obj, ox, oy, and the return address depending on layout) with no bound on cnt before the write.
    - **Fix applied:** Bounded the write: `if (s->has_point(...) && cnt < 100) { list[cnt++] = obj; }`.
    - **Verification:** Confirmed against live source before fixing. `Exult.vcxproj` (Debug|Win32) builds clean (exit 0, zero errors) with the fix in place.

- [x] **[mapedit/shapegroup.cc:765](mapedit/shapegroup.cc#L765)** - ~~del_group() deletes a Shape_group object but never removes its pointer from Shape_group_file::groups~~ **FALSE POSITIVE — no code change made**
    - **Original claim:** del_group() calls `delete groups->get(row)` without erasing the vector entry, leaving a dangling pointer for later use-after-free/double-free.
    - **Why it's not a bug:** The original audit missed that `gtk_tree_store_remove()` (the very next line) synchronously fires GTK's `"row-deleted"` signal, wired in `setup_groups()` to `ExultStudio::groups_changed()`, which calls `groups->remove(row, false)` — erasing the vector entry (without a second delete, since `del=false`) using the same row index, in the same call stack, before any other code can observe a dangling pointer. This signal-driven vector/tree sync is an established pattern used elsewhere in the same file (e.g. reordering). The manual `delete` + signal-driven erase together are correct and equivalent to `Shape_group_file::remove()`.
    - **What happened:** An automated fix (`groups->remove(row)`, erase+delete) was applied and initially looked correct in isolation, but a follow-up adversarial review traced the GTK signal wiring and caught that it causes a **second** `groups->remove()` call on the already-shrunk vector when the signal fires — out-of-bounds on the last row, or silent corruption of an adjacent group otherwise. The change was reverted; the original code is correct as written.
    - **Verification:** Traced `setup_groups()` (signal connect/block/unblock), `groups_changed()`, `on_group_list_row_deleted()`, and `Get_tree_row()` directly to confirm the signal path and index consistency.

- [x] **[server/objserial.cc:50](server/objserial.cc#L50)** - Serial_in::operator<<(std::string&) reads an untrusted 4-byte length and copies that many bytes from the input buffer with no bounds check against the actual received data length `[buffer-overflow]` **[hand-verified]** — **FIXED, build-verified**
    - **Problem:** Object_in/Container_in/Egg_object_in/Npc_actor_in decode network/pipe data of only `datalen` bytes. A corrupted/malicious `len` value (e.g. 0xFFFFFFFF) makes `s.assign(buf, len)` read far past the end of the (typically ≤818-byte) buffer.
    - **Fix applied:** Gave `Serial_in` a real `end` pointer (constructor now takes `(buf, end)`); the string reader clamps the untrusted length to `min(len, end - buf)` before `assign()`, following the same clamp-not-throw idiom already used by `files/databuf.h`'s `IBufferDataView::read()`. Updated all 9 `Serial_in` construction call sites across `objserial.cc` (6), `usecode/ucserial.cc`, `server/server.cc`, and `mapedit/combo.cc` (a 9th site the original audit missed, in the map editor's combo-box deserialization — required since it uses the same class whose constructor changed).
    - **Known residual scope (not fixed here, tracked separately):** an adversarial review found the length prefix itself (`len`) is still read via an unbounded generic template read, so up to 3 bytes can still be over-read before the new clamp applies; and every other primitive field/loop in `Serial_in` (e.g. `Npc_actor_in`'s attacker-controlled `num_schedules` loop) remains completely unchecked. This is the same class of issue already tracked as the **HIGH** finding at `server/objserial.cc:38` below — left for that follow-up rather than expanding this fix's scope.
    - **Verification:** `Exult.vcxproj` (Debug|Win32) builds clean with the new constructor signature and all in-project call sites updated (a missed call site would have been a compile error). `combo.cc` lives in `exult_studio.vcxproj`, also build-verified.

- [x] **[shapeid.cc:587](shapeid.cc#L587)** - Stack buffer overflow (off-by-one write) in Get_palette_transform_table's RampRemap handling `[buffer-overflow]` **[hand-verified]** — **FIXED, build-verified**
    - **Problem:** `from` is derived from the shape's palette_transform value, bounds-checked only to be <= 31 (or ==255). When `from == 31`, the loop writes `remaps[32]`, one past the end of the 32-element stack array `int remaps[32]`.
    - **Fix applied (improved over the audit's suggestion):** Rather than adding a second bounds check inside the loop body (which would silently clobber the caller's requested remap for `from==31` instead of rejecting it — the terminator slot at index 31 is immediately overwritten by an unconditional `remaps[31] = -1;` right after the loop), widened the function's own existing guard clause: `from > std::size(remaps) - 1` → `from >= std::size(remaps) - 1`, so `from==31` is cleanly rejected (returns nullptr) like every other invalid value, consistent with how callers already handle a null transform table.
    - **Verification:** `Exult.vcxproj` (Debug|Win32) builds clean with the fix in place.

- [x] **[tools/wuc.cc:86](tools/wuc.cc#L86)** - strcpy into fixed 10-byte label buffer from an up-to-25600-byte token causes a global buffer overflow. `[buffer-overflow]` **[hand-verified]** — **FIXED, build-verified**

    - **Fix applied (improved over the audit's suggestion):** The audit suggested `strncpy` truncation, but this file's own convention (used 3x in `add_label()`/`read_token()`) is to print an error and `exit()` rather than silently truncate — truncation would also break `get_label()`'s label lookup (full untruncated token compared against a truncated stored name), silently producing garbage jump/data offsets in compiled usecode.     - **Verification:** Standalone `cl.exe` compile of `tools/wuc.cc` (not part of the VS solution; compiled directly with the same headers/flags the real build would use, plus the project's MSVC compat shim for `strcasecmp`) succeeds with zero errors.
    - **Related, out of scope:** `main()` has an identical unbounded-copy pattern at line ~212 (`strcpy(curlabel, token)` into a 256-byte buffer) — not part of this finding, worth a follow-up.

- [x] **[usecode/ucinternal.cc:1613](usecode/ucinternal.cc#L1613)** - Off-by-one bounds check allows out-of-bounds read of the intrinsic function table, leading to an indirect call through a garbage function pointer `[buffer-overflow]` **[hand-verified]** — **FIXED, build-verified**

- [x] **[audio/VocAudioSample.cc:61](audio/VocAudioSample.cc#L61)** - VOC parser reads multiple bytes past the end of the sample buffer without validating that enough bytes remain, for both the header scan and per-frame decompression. `[buffer-overflow]` **CONFIRMED** — **FIXED, build-verified**
    - **Problem:** A truncated or malformed .voc resource (corrupted install, custom mod, or crafted sfx replacement) whose last chunk-type byte sits near size_ causes the loop to read buffer[data_offset], buffer[1+data_offset], buffer[2+data_offset] past the end of the heap allocation (buffer_limit == size_, no slop is ever allocated even though TRAILING_VOC_SLOP/LEADING_VOC_SLOP macros are defined but never used). The same unchecked indexing exists in VocAudioSample::advanceChunk (lines 236-278), which also lets attacker-controlled 24-bit chunk lengths (`l`) push decomp->pos arbitrarily far past buffer_limit; because decompressFrame/advanceChunk only test `decomp->pos == buffer_limit` (not >=), pos can overshoot and subsequent calls keep reading/copying from out-of-bounds heap memory via `std::memcpy(samples, buffer.get() + decomp->pos, num_samples)` (line 323) or the ADPCM decoder (line 329), producing a heap buffer over-read/info leak or crash.
    - **Fix applied:** Added a bounds check before every multi-byte header read in the constructor (all 4 chunk-type cases: Sound data, Sound continues, Silence, Other), bailing out to `last_chunk = true` when not enough bytes remain. In `advanceChunk`, changed the end-of-stream check to `decomp->pos >= buffer_limit` (was `==`) and clamped `chunk_length` to `buffer_limit - decomp->pos` for the "Sound data"/"Sound continue" cases (this also incidentally neutralizes the pre-existing `l -= 2` underflow for tiny declared lengths). `decompressFrame`'s memcpy/ADPCM reads are bounded transitively since `chunk_remain` can no longer imply more bytes than the buffer actually has.
    - **Verification:** `Exult.vcxproj` (Debug|x86) builds clean as part of a full-solution build with zero errors.

- [x] **[audio/midi_drivers/XMidiFile.cpp:1148](audio/midi_drivers/XMidiFile.cpp#L1148)** - Out-of-bounds read of VolumeCurve (128-entry heap array) using an unmasked MIDI data byte that can be up to 255 `[buffer-overflow]` **CONFIRMED** — **FIXED, build-verified**
    - **Problem:** ConvertEvent() reads `data = source->read1()` directly from the XMI/MIDI file with no masking to the 7-bit MIDI data range (0-127). For a Controller-7 (volume) event, current->data[1] is set from this raw byte and is `unsigned char` (0-255).     - **Verification:** `Exult.vcxproj` (Debug|x86) builds clean as part of a full-solution build with zero errors.

- [x] **[mapedit/chunklst.cc:1129](mapedit/chunklst.cc#L1129)** - delete/delete[] mismatch: chunk buffers allocated with array-new are freed with scalar delete `[undefined-behavior]` **CONFIRMED** — **FIXED, build NOT verified (see note)**
    - **Problem:** Every element of `std::vector<unsigned char*> chunklist` is allocated via `new unsigned char[chunksz]` (chunklst.cc:158) or `new unsigned char[datalen]` (chunklst.cc:211), but is released with plain `delete` instead of `delete[]` in three places: the destructor (~Chunk_chooser, line 908), end_terrain_editing() (line 948), and delete_response() (line 1129, triggered whenever the server confirms a terrain/chunk deletion in the map editor).     - **Fix applied:** Changed all three `delete chunklist[...]` call sites (lines 908, 948, 1129) to `delete[] chunklist[...]`.
